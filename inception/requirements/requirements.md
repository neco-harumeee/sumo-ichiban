# 要件定義書 — お相撲さんと一緒

## インテント分析サマリー

| 項目 | 内容 |
|------|------|
| **ユーザーリクエスト** | AWS ハッカソン向け「人をダメにする」サービス。クレーム電話・会議・相談など"ちょっと嫌な場面"に AI のお相撲さんが同席し、精神的に支え続けることで依存を生む |
| **リクエストタイプ** | 新規プロジェクト（Greenfield） |
| **スコープ** | 複数コンポーネント（Web UI + AI エージェント + 音声認識 + 音声生成） |
| **複雑度** | Moderate（AWS サービス複数連携、ハッカソン期間内に完結） |

---

## サービス概要

**お相撲さんと一緒** は、仕事のクレーム電話・気の重い会議・言いづらい相談など「ちょっと嫌な場面」に、Amazon Bedrock を中心とした AI のお相撲さんが同席するサービス。

問題は解決しない。ただ、横で励ましてくれる。

利用者は安心するが、次第に「お相撲さん無しでは電話できない」状態になっていく。

---

## 機能要件

### FR-01: 場面選択 UI
- ユーザーが Web アプリ上で同席してほしい場面を選択できる
  - クレーム電話
  - 上司との面談
  - Zoom 会議
  - 苦手な相談
  - 気まずい報告
- シンプルなハッカソンデモ用画面（認証なし）
- 選択後、AI お相撲さんが起動する

### FR-02: クレーム電話同行モード
- 通話中の緊張を検知すると励ましメッセージをリアルタイム表示
  - 🍵「落ち着いております」
  - 🧂「まわしは取られておりません」
  - 🍲「ここは耐えどころですな」
- Amazon Transcribe でリアルタイム音声認識・緊張ワード検知
- Amazon Bedrock が状況に応じた励ましテキストを生成

### FR-03: 会議同席モード
- Zoom 等の会議画面の隅に常駐表示
- 以下を検知して励ましメッセージを表示
  - 沈黙（一定時間の無音）
  - 発話速度の上昇（Transcribe の文字起こし結果から早口を検知）
  - 長時間ミュート
  - 緊張ワードの頻出（「えーと」「あの」「すみません」等）
- メッセージ例：
  - 🍵「どすこい…」
  - 🍲「長丁場ですな」
  - 🧂（塩を撒くアニメーション）

### FR-04: 依存度トラッキング
- 利用回数・利用時間・依存スコアを記録
- 「お相撲さん依存度」をダッシュボードに表示
- 依存度が上がるほど「もう離れられませんな」等のメッセージを表示

### FR-05: お相撲さん音声
- Amazon Polly でお相撲さんの励まし音声（MP3）を生成
- 緊張検知時に音声を再生
- 音声は S3 に一時保存

### FR-06: AI 励まし生成
- Amazon Bedrock（Claude 3 系）が状況に応じた励ましテキストを生成
- 相撲用語・力士らしい言い回しを使用
- 文脈（クレーム電話 / 会議 / 相談）に応じてトーンを調整

## 非機能要件

### NFR-01: AWS サービス構成
- **Amazon Bedrock** — 励ましテキスト生成（Claude 3 系）
- **Amazon Transcribe** — リアルタイム音声認識・緊張ワード検知・沈黙検知・発話速度解析
- **Amazon Polly** — お相撲さん音声合成（MP3 出力）
- **AWS Lambda** — サーバーレス処理基盤
- **Amazon API Gateway** — Web UI からのリクエスト受付（REST + WebSocket）
- **Amazon S3** — 生成した音声ファイルの一時保存（TTL: 1時間）
- **Amazon DynamoDB** — セッション状態・依存度スコア管理

### NFR-02: パフォーマンス・可用性
- **音声応答レイテンシ**: 緊張検知から励ましメッセージ表示まで **3秒以内**を目標
- **Polly 音声生成**: リクエストから再生開始まで **5秒以内**
- **可用性**: ハッカソンデモ期間中の稼働を確保（SLA 定義なし・PoC）
- **同時接続数**: デモ想定 **最大10セッション**（スケールアウト設計は不要）
- **Lambda タイムアウト**: Orchestrator 30秒 / Analysis 60秒（Transcribe ストリーミング）

### NFR-03: セキュリティ・プライバシー
- 認証なし（ハッカソンデモ用 PoC）
- **会話データは保存しない** — Transcribe の文字起こし結果はメモリ内処理のみ、DynamoDB には保存しない
- **音声ファイルの TTL** — S3 に保存する Polly 生成 MP3 は **1時間後に自動削除**（S3 Lifecycle）
- **個人情報の非永続化** — 氏名・連絡先等の個人情報は一切保存しない
- **DynamoDB 保存データ** — セッション ID・場面・時間・励まし回数・依存度スコアのみ（発話内容は含まない）
- セキュリティ拡張ルール: **スキップ**（PoC のため）

### NFR-04: コスト制約
- **Bedrock** — Claude 3 Haiku を優先使用（Sonnet はフォールバック）。1セッションあたりの推定コスト: **$0.01 以下**
- **Transcribe** — ストリーミング利用。デモ期間の総利用時間: **10時間以内**を目安
- **Polly** — 標準音声を使用。1リクエストあたり最大500文字
- **月間コスト上限目安**: **$50以下**（ハッカソンデモ期間）
- **CloudWatch アラーム**: Bedrock / Transcribe の利用コストが閾値を超えた場合に通知

### NFR-05: 運用性・障害時の挙動
- **CloudWatch Logs** — 全 Lambda のログを記録（エラー・レイテンシ）
- **Bedrock 応答失敗時** — プリセットの励ましメッセージ（固定文言）にフォールバック
- **Transcribe 接続断時** — フロントエンドに「音声認識を再接続中」を表示し、自動リトライ（最大3回）
- **Lambda タイムアウト時** — エラーレスポンスを返し、フロントエンドは次の検知イベントを待機
- **デプロイ**: AWS CDK による再現可能なデプロイ（`cdk deploy` 1コマンド）

### NFR-06: テスト方針
- プロパティベーステスト: **スキップ**（PoC のため）
- 基本的な動作確認テストのみ実施

### NFR-07: デモ・提供形態
- Web アプリ（ブラウザ動作、PC・スマホ対応）
- デモ形式: スライド + 動画デモ（録画）
- ライブデモは不要

### NFR-08: 開発スコープ
- ハッカソン期間内に完結する実装量
- コア機能（FR-01〜FR-06）を実装、一部モック実装で工数を抑える

## 成功基準

- Web アプリ上で場面選択 → お相撲さん同席 → 緊張検知 → 励ましメッセージ表示 → 音声再生 の一連のフローがデモ動画で確認できる
- AWS サービス（Bedrock / Lambda / Transcribe / Polly / API Gateway / S3）を実際に使用している
- ハッカソン審査員に「笑えないリアリティ」と「技術的完成度」が伝わる
- 依存度スコアが上昇していく様子がデモで確認できる

## KPI / Success Metrics

### ユーザー価値の指標

| KPI | 目標値 | 計測方法 |
|-----|--------|---------|
| 1セッション平均時間 | **5分以上** | DynamoDB の duration_sec 平均 |
| セッション完走率（途中離脱なし） | **80%以上** | end_session 呼び出し率 |
| リピート利用率（3日以内に再利用） | **50%以上** | DynamoDB の last_session_at 差分 |
| 依存度スコア上昇ユーザー割合 | **70%以上** | dependency_scores テーブルの score 推移 |

### AI・システム品質の指標

| KPI | 目標値 | 計測方法 |
|-----|--------|---------|
| 音声認識成功率 | **90%以上** | Transcribe エラーレート |
| 励まし応答レイテンシ | **3秒以内** | CloudWatch Lambda Duration |
| Bedrock 応答失敗率 | **5%以下** | Lambda エラーログ |
| フォールバック発動率 | **10%以下** | プリセット応答の使用回数 |

### ハッカソン評価軸との対応

| 評価軸 | 対応する指標 |
|--------|------------|
| ビジネス意図の明確さ | 依存度スコア上昇ユーザー割合 |
| 創造性とテーマ適合性 | 「笑えないリアリティ」の審査員反応 |
| 技術的完成度 | 応答レイテンシ・音声認識成功率 |

### 仮説検証（MVP で学ぶこと）

- **仮説1**: ユーザーは「会話内容の正確さ」より「そこにいる存在感」に価値を感じる
  - 検証: 励ましメッセージの内容より音声再生のオン/オフで継続率が変わるか
- **仮説2**: 音声（Polly）はテキストより依存度を高める
  - 検証: 音声あり/なしセッションの平均時間を比較
- **仮説3**: 依存度の可視化がリピート利用を促進する
  - 検証: ダッシュボード閲覧後の再利用率

## リスク・倫理・ガードレール

### 倫理リスク

| リスク | 対策 |
|--------|------|
| 孤独感への過度な依存 | 依存度「横綱」到達時に「たまには一人で挑戦してみましょう」を表示（逆説的メッセージ） |
| メンタルヘルス悪化の助長 | センシティブワード（「死にたい」「消えたい」等）を検知した場合は固定応答に切り替え、励ましを停止 |
| 不適切な励まし生成 | Bedrock の出力に不適切語句が含まれる場合はフォールバックメッセージに差し替え |

### プライバシーリスク

| リスク | 対策 |
|--------|------|
| 会話内容の漏洩 | Transcribe の文字起こし結果は DynamoDB に保存しない（メモリ内処理のみ） |
| 音声データの残存 | S3 の Polly 生成 MP3 に TTL 1時間を設定（S3 Lifecycle） |
| 個人情報の蓄積 | 氏名・連絡先等は一切収集・保存しない |

### AI 誤判定リスク

| リスク | 対策 |
|--------|------|
| 緊張ワードの誤検知 | 検知閾値を調整可能にする（デフォルト: 同一ワード2回以上） |
| 沈黙の誤検知（背景音） | 3秒以上の連続無音のみ検知（短い間は無視） |
| Bedrock の不適切生成 | システムプロンプトで相撲用語・励ましのみに制約。出力フィルタリングを実装 |

### コスト暴走リスク

| リスク | 対策 |
|--------|------|
| Bedrock 過剰呼び出し | 同一セッション内で30秒以内の連続呼び出しをスロットリング |
| Transcribe ストリーミング放置 | セッション終了時に必ず Transcribe ストリームを停止。Lambda タイムアウトでも強制終了 |
| コスト超過 | CloudWatch アラームで月額 $50 超過時に SNS 通知 |
